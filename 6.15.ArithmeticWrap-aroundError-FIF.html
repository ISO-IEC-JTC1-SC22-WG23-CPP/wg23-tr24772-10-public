6.15 Arithmetic Wrap-around Error [FIF]

6.15.1 Applicability to language

C++ has the vulnerability as documented in ISO/IEC TR 24772-1:2019 clause 6.15. C++ specifies that * signed integers are defined to be represented as two’s complement, still signed integer overflow is undefined behaviour, * unsigned integer overflow is well-defined in C++ to wrap-around, * integral promotion can change values of unsigned types to a signed type before arithmetic takes place, mixing signed and unsigned values or integral types of different conversion rank in an expression is subject to the implicit “usual arithmetic conversions” that can be confusing, * integral values implicitly convert among all integral types, even when passed as a function argument, and, * using brace-initialization prevents implicit narrowing conversions.

Even the unsigned wrap around can be confusing, such as 4U - 5U yields a large positive value. Multiplying two unsigned short variables, each with a large value, can result in undefined behavior. For example:

auto f(){
  std::uint16_t x{50'000},y{50'000};
  return x * y; // undefined behaviour due to overflow, returns int
}

In the above, for a 16-bit short and a 32-bit int, i.e., std::numeric_limits<int>::max()==0x7fff'ffff, x and y are promoted to int and the multiplication then overflows which is undefined behaviour.

Calling a function taking a parameter of integral type with an argument of mismatched integral type works due to implicit conversion. If a different overload with a better match becomes visible the called function can change when re-compiled (see 6.21 Namespace Issues[BJL])

The mitigations for wrap-around errors in C++ are different than for C. The type system of C++ allows user-defined class and enum types with corresponding overloaded operators. Such user-defined types can individually control which implicit conversions or mixed type arithmetic they support, if any. For example, one can force arithmetic to be done with unsigned types:

enum class uint16: std::uint16_t{};
uint16 operator*(uint16 a, uint16 b){
  return static_cast<uint16>(static_cast<unsigned>(a) * static_cast<unsigned>(b));
} // guarantee wrap around

High-integrity software using the built-in integral types should refrain from relying on integral promotion and implicit conversions and use integral types with a well-known range, or guarantee that allowed variation in the range of representable values do not influence computed results.

6.15.2 Avoidance mechanisms for language users

  • Use the avoidance mechanisms of ISO/IEC 24772-1 clause 6.15.5.

  • Use appropriate user-defined types in place of built-in integral types, that mitigate against implicit conversions and undefined behaviour.

  • Avoid implicit integral promotions and integral conversions, especially on function arguments.

  • Avoid mixing integral types in the same expression.

  • Ensure that the result of any mathematical operation fits within the constraints of the types involved within the expression.

  • If you intend arithmetic to wrap, use an unsigned type that won’t promote to int.

  • Consider the use of numeric_limits<T>::is_modulo to determine whether or not an integer type T wraps for the target system.

  • Document where wrap-around is expected for a type.
